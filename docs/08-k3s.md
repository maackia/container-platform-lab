# K3s 기반 Kubernetes 배포

이 문서는 Docker Compose로 구성한 애플리케이션 스택을 단일 노드 K3s 환경으로 옮긴 과정과 운영 명령을 정리한다.

구성 범위는 다음과 같다.

```text
Traefik Ingress
→ app Service
→ Node.js Deployment
→ db Service
→ PostgreSQL StatefulSet
→ PersistentVolumeClaim
```

## 1. 실습 환경

```text
Host: Apple Silicon Mac
VM: UTM Ubuntu Server 24.04 ARM64
Kubernetes: K3s 단일 노드
Ingress Controller: K3s 기본 Traefik
Container Registry: GHCR
Namespace: platform-lab
```

K3s 설치 후 노드 상태를 확인한다.

```bash
kubectl get nodes
kubectl get pods -A
```

K3s의 기본 kubeconfig를 일반 사용자 계정에서 사용하려면 복사본을 만든다.

```bash
mkdir -p ~/.kube
sudo cp /etc/rancher/k3s/k3s.yaml ~/.kube/config
sudo chown "$USER":"$USER" ~/.kube/config
chmod 600 ~/.kube/config
export KUBECONFIG="$HOME/.kube/config"
```

기존 `KUBECONFIG` 환경변수가 `/etc/rancher/k3s/k3s.yaml`을 가리키고 있으면 복사 후에도 권한 오류가 발생할 수 있다. 이때는 위와 같이 사용자 설정 파일을 명시하거나 환경변수를 해제한다.

매번 namespace 옵션을 입력하지 않으려면 현재 context의 기본 namespace를 설정할 수 있다.

```bash
kubectl config set-context --current --namespace=platform-lab
kubectl config view --minify | grep namespace
```

## 2. 매니페스트 구조

```text
k8s/
├── 00-namespace.yaml
├── 01-nginx-deployment.yaml
├── 02-nginx-service.yaml
├── 03-app-deployment.yaml
├── 04-app-service.yaml
├── 05-platform-configmap.yaml
├── 06-db-init-configmap.yaml
├── 07-db-service.yaml
├── 08-db-statefulset.yaml
└── 09-app-ingress.yaml
```

파일명 앞의 번호는 적용 순서를 이해하기 위한 구분이다. Kubernetes는 선언된 리소스 상태를 기준으로 동작하므로 반복 적용해도 필요한 차이만 반영한다.

```bash
kubectl apply -f k8s/
```

## 3. Namespace와 Nginx 기본 실습

모든 리소스는 `platform-lab` namespace에서 관리한다.

```bash
kubectl get all -n platform-lab
```

Nginx Deployment는 replica 두 개와 readiness/liveness probe, 리소스 요청·제한을 정의한다. NodePort Service는 다음 경로를 만든다.

```text
Ubuntu VM:30080
→ nginx Service:80
→ nginx Pod:80
```

Pod 하나를 삭제해도 Deployment가 원하는 replica 수를 다시 맞추는지 확인할 수 있다.

```bash
kubectl get pods -l app=nginx
kubectl delete pod <NGINX_POD>
kubectl get pods -l app=nginx -w
```

## 4. GHCR 애플리케이션 배포

애플리케이션 Deployment는 GHCR에 게시한 멀티 아키텍처 이미지를 사용한다.

```text
ghcr.io/maackia/container-platform-lab-app:0.1.0
```

이 이미지는 `linux/amd64`와 `linux/arm64`를 함께 지원하므로 ARM64 Ubuntu VM에서도 별도 플랫폼 지정 없이 실행된다.

애플리케이션은 replica 두 개로 실행되며 `/health`를 readiness와 liveness probe에 사용한다. ClusterIP Service를 통해 클러스터 내부에서 `app:3000`으로 접근한다.

```bash
kubectl get deployment app
kubectl get pods -l app=platform-app
kubectl get service app
kubectl get endpointslice -l kubernetes.io/service-name=app
kubectl rollout status deployment/app
```

Kubernetes v1.33 이후에는 기존 Endpoints API가 deprecated 상태이므로 신규 확인 명령에는 EndpointSlice를 사용한다.

## 5. ConfigMap과 Secret

일반 설정은 `platform-config` ConfigMap으로 관리한다.

```text
DB_HOST=db
DB_PORT=5432
DB_USER=appuser
DB_NAME=appdb
POSTGRES_USER=appuser
POSTGRES_DB=appdb
```

비밀번호는 Git에 커밋하지 않고 클러스터에 Secret으로 생성한다.

```bash
LAB_DB_PASSWORD="$(openssl rand -hex 16)"

kubectl create secret generic platform-secret \
  --namespace platform-lab \
  --from-literal=DB_PASSWORD="$LAB_DB_PASSWORD" \
  --from-literal=POSTGRES_PASSWORD="$LAB_DB_PASSWORD" \
  --dry-run=client -o yaml | kubectl apply -f -

unset LAB_DB_PASSWORD
```

Secret 존재 여부만 확인하고 실제 값을 터미널이나 문서에 출력하지 않는다.

```bash
kubectl get secret platform-secret -n platform-lab
```

## 6. PostgreSQL StatefulSet와 영속 저장소

PostgreSQL은 고정된 Pod 이름과 저장소 연결이 필요한 상태 저장 워크로드이므로 StatefulSet으로 구성했다.

```text
db headless Service
→ db-0 Pod
→ db-data PVC
→ K3s local-path StorageClass
```

주요 구성은 다음과 같다.

```text
StatefulSet replica: 1
Service DNS: db
Container image: postgres:17-alpine
StorageClass: local-path
PVC size: 2Gi
Data path: /var/lib/postgresql/data
```

`db-init` ConfigMap의 `init.sql`은 데이터 디렉터리가 비어 있는 최초 초기화 시점에만 실행된다.

상태를 확인한다.

```bash
kubectl get statefulset db
kubectl get pod db-0
kubectl get service db
kubectl get pvc
kubectl describe pvc db-data-db-0
```

Pod를 다시 만들어도 PVC가 유지되는지 검증한다.

```bash
kubectl delete pod db-0
kubectl get pod db-0 -w
kubectl get pvc
```

## 7. Traefik Ingress

K3s가 기본 제공하는 Traefik을 Ingress Controller로 사용한다.

`platform.local` 요청은 다음 경로로 전달된다.

```text
Host header: platform.local
→ Traefik
→ app Service:3000
→ Node.js Pod:3000
```

Ingress 상태를 확인한다.

```bash
kubectl get ingress
kubectl describe ingress app
kubectl get pods -n kube-system -l app.kubernetes.io/name=traefik
```

UTM의 Emulated VLAN을 사용하면 Mac에서 VM의 80번 포트로 포트 포워딩을 추가한다. 예를 들어 Mac의 8082번을 게스트 80번으로 연결한 경우 다음처럼 확인할 수 있다.

```bash
curl -H 'Host: platform.local' http://localhost:8082/health
curl -H 'Host: platform.local' http://localhost:8082/
```

브라우저에서 `http://platform.local:8082`를 사용하려면 Mac의 `/etc/hosts`에 다음 항목을 추가한다.

```text
127.0.0.1 platform.local
```

호스트 포트는 UTM 설정에 따라 다를 수 있다.

## 8. 매니페스트 검증

클러스터에 적용하기 전 두 종류의 검증을 사용한다.

### 서버 측 dry-run

현재 클러스터의 API와 admission 설정을 기준으로 검사한다.

```bash
kubectl apply --dry-run=server -f k8s/
```

### Kubeconform

CI에서는 Kubeconform `v0.8.0`을 고정해서 정적 스키마를 검사한다.

```bash
docker run --rm \
  -v "$PWD:/work" \
  ghcr.io/yannh/kubeconform:v0.8.0 \
  -strict -summary /work/k8s
```

`latest` 대신 검증한 버전을 고정하면 같은 커밋이 실행 시점과 관계없이 동일한 도구 버전으로 검사되어 CI 재현성이 좋아진다. 버전을 올릴 때는 릴리스와 호환성을 확인한 뒤 명시적으로 변경한다.

두 검사는 역할이 다르다.

```text
Kubeconform
→ CI에서 빠른 스키마·알 수 없는 필드 검사

kubectl --dry-run=server
→ 실제 K3s API 기준 검사
```

## 9. 운영 확인 명령

```bash
kubectl get all
kubectl get ingress,pvc,configmap,secret
kubectl rollout status deployment/app
kubectl logs deployment/app --tail=100
kubectl logs statefulset/db --tail=100
kubectl describe pod <POD_NAME>
kubectl get events --sort-by=.lastTimestamp
```

다른 namespace를 확인할 때는 `-n <NAMESPACE>`를 명시한다. 현재 context의 기본 namespace를 변경했더라도 CI나 문서의 재현 가능한 명령에는 namespace를 명시하는 편이 안전하다.

## 10. Docker Compose와 Kubernetes 대응

| Docker Compose | Kubernetes |
|---|---|
| service | Deployment 또는 StatefulSet |
| service name DNS | Service DNS |
| ports | Service와 Ingress |
| environment | ConfigMap과 Secret |
| named volume | PersistentVolumeClaim |
| healthcheck | readiness/liveness probe |
| restart policy | Controller의 상태 조정 |
| compose.yaml | Kubernetes 매니페스트 묶음 |

Compose는 단일 호스트에서 애플리케이션 구조를 빠르게 실행하기 좋다. Kubernetes는 선언한 상태를 컨트롤러가 지속적으로 맞추고, Service·Ingress·스토리지를 독립된 리소스로 관리한다.

## 11. 현재 완료 범위

```text
K3s 단일 노드 설치와 kubeconfig 구성
→ Namespace 분리
→ Nginx Deployment와 NodePort Service
→ GHCR 애플리케이션 Deployment
→ ConfigMap과 Secret 분리
→ PostgreSQL StatefulSet
→ local-path PVC 영속성
→ Traefik Ingress
→ readiness/liveness probe
→ Kubeconform CI 검증
→ 서버 측 dry-run 검증
```

다음 확장 단계에서는 Kubernetes 환경의 메트릭 수집과 대시보드 구성을 다룬다.
